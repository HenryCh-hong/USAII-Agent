# Human-in-the-Loop Design

Forked Futures is built so the human stays in control at every step. The system
maps the futures; the person makes the call.

## The human drives the pipeline

- **Intake** — the user describes the decision, options, values, constraints and
  fears in their own words.
- **Clarify** — the user answers AI-generated clarifying questions (or uses the
  demo answers); the system never proceeds on assumptions it hasn't surfaced.
- **Explore** — the user opens each branch, reads its evidence, assumptions, and
  pre-mortem, and talks to a branch-grounded "Future Self".
- **Decide** — the Decision Brief frames what is really being decided and hands the
  choice back to the user.

## What the AI explicitly will not decide

The Decision Brief contains a dedicated panel listing the human-only calls:

- how to weight competing values against each other,
- the user's true risk tolerance and personal context,
- which kind of regret the user could live with.

These turn on values, risk tolerance, family, identity and lived experience —
context no model holds. Every results surface repeats the same framing: these are
plausible scenarios, not predictions, and the final decision stays with the user.

## Designed to be tested, not obeyed

Instead of an answer to accept, each branch ends in a **7-day experiment** with
pre-committed kill criteria — a cheap, reversible way to replace an assumption with
real-world signal before committing. The point is not to follow the system; it is
to act on better information.

## Transparency for trust

A Judge Mode / Architecture page shows the full pipeline, the official-source
evidence pack, the local evidence graph, the 9-role debate, the safety scrubber,
the reasoning audit trail, and a live sample trace of the Alex demo — so a person
(or a judge) can see exactly how a conclusion was assembled and where the human
remains in control.

## Calibrated, reversible, honest

- Calibration is qualitative, so the user is never anchored on a false precise number.
- Reversibility is tagged per branch, so deliberation can match how hard a choice is to undo.
- Assumptions are labelled by provenance and paired with a way to test them, so the
  user can tell what is known from what is inferred.
